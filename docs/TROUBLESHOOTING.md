# 트러블슈팅 로그

이 프로젝트를 실제 데이터로 검증하는 과정에서 부딪힌 문제들과 해결 과정을 기록합니다.
"이상탐지 로직을 짰다"보다 "실전에서 뭐가 깨졌고 왜 깨졌는지 파악해서 고쳤다"는 과정 자체가
이 프로젝트의 핵심 자산이라고 생각해서, 발생 순서대로 정직하게 남깁니다.

---

## 1. 데이터셋 자체가 잘못됨 — Kaggle → NOAA 전환

**증상**: Kaggle "AIS Dataset"(eminserkanerdonmez)을 다운로드해서 열어보니 위도/경도, 시간 컬럼이
아예 없었음. 컬럼은 `mmsi, navigationalstatus, sog, cog, heading, shiptype, width, length, draught`뿐.

**원인**: Kaggle 데이터셋 상세페이지 설명 텍스트만 보고 실제 컬럼 구조를 확인하지 않은 채 계획을 세움.
실제 배포된 CSV는 정적 속성 위주의 축약 버전이었음.

**해결**: 위치 기반 이상탐지가 프로젝트 핵심이라 위경도/시간이 없으면 성립 자체가 안 됨 → 대안 데이터
소스 탐색 → **NOAA MarineCadastre 공식 공개 데이터**(coast.noaa.gov)로 전환. 로그인 불필요, 스키마
사전 검증(MMSI, BaseDateTime, LAT, LON, SOG, COG, Heading 등) 후 진행.

**교훈**: 데이터셋을 정하기 전에 실제 파일을 최소 몇 줄이라도 열어서 스키마를 확인해야 함. 설명 텍스트는
신뢰할 수 있는 근거가 아님.

## 2. DBSCAN 메모리 폭발

**증상**: NOAA 데이터(SF Bay 필터링, 약 17만행)로 `extract_waypoints()`를 돌리자 프로세스가 `Killed`로
강제 종료됨.

**원인 조사**: 단계적으로 행 수를 늘려가며(2만 → 5만 → 8만 → 10만) 재현 시도 → 10만행에서
`resource.getrusage()`로 측정한 최대 메모리가 **3.2GB**로 확인됨. 원인은 haversine 거리 기준
DBSCAN이 정박지처럼 극도로 밀집된 구역에서, ball_tree 알고리즘이라도 한 포인트 주변에 수천 개의
이웃이 몰리는 경우 메모리 사용량이 급격히 늘어나는 구조적 특성 때문.

**해결**: waypoint 학습은 전체 포인트의 밀도 패턴만 알면 충분하고 모든 포인트가 필요한 게 아니므로,
`extract_waypoints()`에 `max_points` 파라미터를 추가해 최대 3만 포인트로 샘플링 후 학습하도록 수정.
(`src/detection/routes.py`)

**결과**: 17만행 전체를 그대로 넣어도 0.9초, 462MB로 정상 처리.

## 3. 정박 선박의 COG 노이즈로 인한 대량 오탐

**증상**: `detect_kinematic_jumps()`를 실제 데이터에 돌리니 급변침로가 **9,569건**(전체의 5.5%)이나
탐지됨. 너무 많아서 의심.

**원인**: 몇 건을 직접 까보니, 정박 중이거나 속도가 거의 0인 선박에서 COG(침로) 값 자체가 GPS 신호
노이즈로 크게 흔들리는 게 정상적인 현상이었음. 배가 실제로 방향을 튼 게 아니라, 가만히 떠있는데
GPS가 침로를 잘못 계산해서 튀는 것.

**해결**: 저속(기본 2노트 미만) 구간에서는 COG 값의 신뢰도가 낮다고 보고, 침로 급변 판정 자체를
건너뛰도록 `min_speed_for_course_check_knots` 조건 추가. (`src/detection/anomaly.py`)

**결과**: 9,569건 → **802건**으로 정상화. 오탐 비율이 실제로 이상한 케이스만 남도록 개선됨.

## 4. 항로이탈 탐지 속도 — 72초 → 0.12초 (약 600배)

**증상**: `detect_route_deviation()`이 17만행 처리에 **72.5초** 소요. 파라미터 튜닝하면서 여러 번
돌려봐야 하는데 매번 1분 이상 걸리는 게 비효율적.

**원인**: 최근접 waypoint 거리를 구할 때 `df.apply()`로 각 행마다 파이썬 루프를 돌며 모든 waypoint와
거리를 일일이 계산하는 구조였음 (O(N×M), N=포인트 수, M=waypoint 수).

**해결**: waypoint 좌표로 **BallTree(haversine)**를 한 번만 구축해두고, 전체 포인트에 대해 벡터화된
`query()`로 한 번에 최근접 거리를 계산하도록 재작성. (`src/detection/routes.py`의
`build_waypoint_tree`, `query_nearest_waypoint_distances`)

**결과**: 72.5초 → **0.12초**. 이후 파라미터 튜닝(threshold_km 스윕 등)을 여러 번 반복해도 부담 없어짐.

## 5. Groq reasoning 모델의 빈 응답

**증상**: 실제 Groq API(`openai/gpt-oss-120b`)로 이상 항적 설명을 배치 생성하던 중, 일부(`implausible_speed`
케이스 1건)에서 `content`가 빈 문자열로 반환됨. 단독으로 재현 시도(4회)했을 땐 재현 안 됨 — 간헐적 이슈.

**원인 조사**: 해당 모델 응답의 reasoning 토큰 사용량을 확인해보니 68~104 토큰 정도를 내부 사고 과정에
소비하고 있었음. `gpt-oss-120b`는 reasoning 모델이라 답변 전에 내부적으로 생각을 하는데, 기존
`max_tokens=300` 설정이 낮아서 reasoning이 길어지면 실제 답변(content)에 배정될 토큰이 부족해져
빈 문자열로 잘리는 구조적 위험이 있었음.

**해결**:
1. `max_tokens`를 300 → **500**으로 늘려 여유 확보 (`src/ai_layer/explainer.py`)
2. 그래도 빈 응답이 올 가능성은 남아있으므로, `content`가 비어있으면 조용히 실패하는 대신
   규칙 기반 폴백 설명으로 자동 대체하는 안전장치 추가
3. 네트워크 없이도 이 로직을 검증할 수 있도록 `unittest.mock`으로 Groq 응답을 모킹하는
   단위테스트 3종 추가 (`tests/test_explainer.py`)

**교훈**: reasoning 모델을 API로 쓸 때는 `max_tokens`가 "답변 길이"가 아니라 "reasoning + 답변 합산
길이"라는 걸 감안해서 여유 있게 잡아야 함. 그리고 LLM 응답은 언제든 예상 밖의 형태(빈 값 등)로 올 수
있다는 전제로 항상 폴백 경로를 만들어둬야 함.

## 6. 대시보드 지도 조작 시 전체 파이프라인 무한 재계산

**증상**: 배포한 Streamlit Cloud 라이브 데모에서 지도를 클릭/드래그하거나 탭을 전환할 때마다
"데이터 정제 중..."부터 다시 시작하고, 이상 리스트/통계 탭에 아무 결과도 안 나타남.

**원인 조사**:
1. `st_folium`(인터랙티브 지도 컴포넌트)은 지도를 팬/줌/클릭할 때마다 그 상호작용 정보를
   Python으로 돌려주기 위해 **스크립트 전체를 재실행**시키는 구조. 이 재실행 자체는 Streamlit의
   정상 동작이지만, 문제는 재실행마다 정제/항로학습/이상탐지를 처음부터 다시 계산했다는 것.
2. 캐싱(`st.cache_data`)이 하나도 적용되어 있지 않아서, 매 재실행마다 전체 파이프라인이
   무거운 계산(당시 급변침로 탐지만 약 20초)을 반복함.
3. Streamlit Cloud 무료 티어는 로컬보다 CPU/메모리가 제한적이라, 이 반복 계산이 로컬보다
   훨씬 느리게 체감되거나 타임아웃/행업처럼 보였을 가능성이 높음.

**해결**:
1. `detect_dark_gaps`, `detect_kinematic_jumps`를 파이썬 반복문에서 pandas
   `groupby().shift()` + numpy 벡터 연산으로 재작성 (급변침로 탐지: **20초 → 0.16초**).
2. 데이터 로딩/정제, waypoint 학습, 이상탐지 3종 전부 `@st.cache_data`로 감싸서, 입력이
   동일하면 재계산 없이 캐시된 결과를 즉시 반환하도록 수정.
3. `st_folium` 호출에 `returned_objects=[]`를 지정해, 굳이 필요 없는 상호작용 데이터까지
   되돌려 받지 않도록 해 불필요한 재실행 트리거를 줄임.

**검증**: 캐싱/벡터화 전후로 신호중단(62건), 급변침로(802건) 탐지 결과가 정확히 동일함을 확인 —
성능만 개선되고 로직 결과는 그대로 유지됨.

## 7. 고정 임계값의 과학적 근거 부족 → 선박종류별 z-score 통계 기반으로 전환

**증상**: 급변침로/속도 탐지가 "속도 40노트 초과", "침로변화 90도 초과"라는 고정 임계값을
모든 선박에 동일하게 적용하고 있었음. 사용자 피드백: "이거 임의로 정한 거 아니야? 과학적
근거가 있었으면 좋겠다."

**원인**: 앞선 검증(선박타입별 교차분석)에서 이미 확인했듯, 예인선/파일럿선은 원래 급선회가
잦고 화물선/유조선은 거의 직선 항로를 유지하는데, 모든 선박에 똑같은 절대 기준(90도)을
적용하면 "그 선박종류에서는 지극히 정상적인 행동"까지 이상으로 잡힐 수 있음.

**해결**: `detect_kinematic_jumps_statistical()` 함수를 추가해, 같은 선박종류 집단 안에서
평균·표준편차 대비 z-score를 계산하도록 전환 (`src/detection/stats.py`의 `group_zscore`).
판단 조건은 (1) 해당 선박종류 평균 대비 |z-score| > 3 이면서 (2) 동시에 물리적으로도 의미
있는 절대 크기(속도 25노트 또는 침로변화 60도 이상)를 모두 만족해야 이상으로 판단. 표본이
적은 선박종류(30척 미만)는 전체 데이터 기준으로 자동 대체(fallback).

**결과**: 화물선/유조선은 여전히 거의 탐지되지 않고(정상 유지), 결과에 z-score와 백분위수가
함께 표시되어 "왜 이상으로 판단했는지"에 구체적 통계 근거가 남음. 다만 VesselType 하위코드를
큰 범주(예: 60~69 전부 "여객선")로 묶다 보니 성격이 다른 선박이 한 그룹으로 묶여 그룹 내
분산이 커지는 부작용도 확인됨 — 그룹 세분화는 추후 개선 여지로 남겨둠.

## 8. waypoint 방향성 부재 & 신호중단 임계값의 "근거 있는 부분 vs 실용적 선택" 구분

**질문**: "DBSCAN으로 waypoint 학습했다는데, 선박마다 경로가 다를 텐데 저렇게 점만 모아놔도 되는 거야?"
"신호중단 60분은 네가 임의로 정한 거 아니야?"

**waypoint 관련 솔직한 답**: DBSCAN은 "포인트가 밀집된 곳"만 waypoint로 잡아낼 뿐, 그 지점들
사이를 어떻게 이었는지(경로/방향)는 전혀 모른다. 목적지가 달라도 항구 입구·좁은 해협·정박지처럼
대부분의 선박이 공통으로 지나가는 지점은 실제로 존재하기 때문에 "1차 근사치"로는 어느 정도
말이 되지만, 입구를 지나 정상적으로 갈라지는 두 항로를 구분하지 못해 정상 항해도 항로이탈로
오탐될 수 있다는 게 진짜 한계다. 더 정확히 하려면 (1) 선박별 궤적을 DTW로 유사도 클러스터링,
(2) TREAD 논문처럼 방향성 있는 흐름(flow) 추출, (3) 격자 기반 밀도맵+방향정보 저장이 필요하다 —
이번 프로젝트는 스코프상 가장 단순한 방식만 구현했다.

**60분 임계값 관련 솔직한 답**: "방향성"과 "정확한 숫자"를 구분해서 답해야 한다.
- 방향성(정상 최댓값보다 훨씬 커야 이상으로 볼 근거가 생김)은 ITU-R M.1371 국제표준(정박 중
  최대 3분 간격 규정)으로 뒷받침됨.
- 하지만 정확히 왜 60분인지(30분이나 45분이 아니라)는 절대적 정답이 없다. "3분의 20배면
  충분히 보수적인 여유값"이라는 실용적 판단으로 정한 반올림된 숫자일 뿐이다.

**교훈**: 하이퍼파라미터나 임계값을 정할 때 "이 숫자가 왜 이 값인지"를 (1) 방향성이 근거 있는
부분과 (2) 정확한 수치는 실용적으로 조정한 부분으로 나눠서 정직하게 설명하는 게, 모든 걸
"과학적으로 완벽하다"고 포장하는 것보다 훨씬 신뢰를 준다. 이렇게 한계를 명확히 남겨두면
나중에 실제로 개선할 지점(percentile 기반 임계값, 궤적 유사도 클러스터링 등)도 분명해진다.



| 문제 | 핵심 교훈 |
|---|---|
| 데이터셋 스키마 미검증 | 계획 전에 실제 데이터를 반드시 열어볼 것 |
| DBSCAN 메모리 폭발 | 학습 목적에 맞게 전체 데이터가 필요한지부터 따질 것 (샘플링으로 충분한 경우가 많음) |
| COG 노이즈 오탐 | 센서 데이터는 물리적 상태(저속/정박)에 따라 신뢰도가 달라진다는 걸 로직에 반영해야 함 |
| 성능 병목 | 파이썬 루프 대신 벡터화(BallTree 등)로 구조적 개선이 가능한지 먼저 검토 |
| LLM 빈 응답 | reasoning 모델의 토큰 예산 특성 이해 + 항상 폴백 경로 설계 |
| 대시보드 무한 재계산 | 인터랙티브 컴포넌트(지도 등)는 재실행을 유발하므로, 무거운 계산은 반드시 캐싱해야 함 |
| 고정 임계값의 근거 부족 | "일괄 기준"보다 "같은 집단 내 상대적 이상치(z-score)"가 더 방어 가능한 판단 근거가 됨 |
| 하이퍼파라미터의 근거 유무 구분 | "방향성은 근거 있음"과 "정확한 숫자는 실용적 선택"을 구분해서 정직하게 명시하는 게 신뢰를 만듦 |
